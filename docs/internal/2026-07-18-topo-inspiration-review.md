# Topo inspiration review: the ruled best-of-breed pass (next-pass brief)

Produced 2026-07-18 at the docs-on-site pipeline pass's close, per the 2026-07-06 ruling
that Topo opens with an inspiration review of internet-beloved docs systems before any
build. Four parallel research reads (VitePress, Fumadocs, Mintlify as polish reference,
and a Starlight anatomy recap) synthesized below; verify popularity figures at design
time. The next pass opens here: Geoff reads section 5's open questions, mockup candidates
go to him BEFORE the build (the ruled gate), and the anatomy checklist in section 4 is
the port's spine. The pipeline Topo inherits is live: cairn.pub renders the docs from the
npm package with TOCs, prev/next, breadcrumbs, and the anchor-compatibility gate, all
built portable for the docs.cairn.pub move.

**VitePress.** VitePress is loved because it collapses the distance between "write Markdown" and "have a polished site" to almost nothing. `npx vitepress init` produces a themed, navigable site in minutes, and the default theme already carries navigation, sidebar, dark mode, local search, edit links, last-updated stamps, and i18n scaffolding without plugin assembly. It inherits Vite's speed, so edit-to-preview is near-instant, which matters to authors making many small content passes. Its credibility comes from who runs it: Vue, Vite, Vitest, Rollup, Pinia, VueUse, and D3 all ship their docs on it, so developers meet VitePress output constantly. The default theme reads deliberately plain, a single centered column with Shiki code blocks doing the visual work, which keeps attention on the content rather than the chrome.

**Fumadocs.** Fumadocs is loved in the Next.js world for feeling like a native surface of the product it documents rather than a bolted-on generator. It is built as three separable layers (content, headless core, Tailwind UI), so a team can take only the search and TOC primitives or eject individual theme components into their own tree in the shadcn ownership model. Reviewers call it "a docs framework you can actually break," meaning deep customization survives upstream updates. Its authoring tooling is unusually deep for the category: type-safe MDX content collections validated by Zod, Twoslash type-checking inside code samples, and tables generated straight from TypeScript types. It grew from about 300 GitHub stars in early 2024 to roughly 12,000 by mid-2026, one of the fastest climbs in the category.

**Mintlify.** Mintlify is the polish reference, loved for output that reads premium out of the box across a concentration of AI and dev-tool customers (Anthropic, Cursor, Perplexity, Resend, ElevenLabs). The premium feel is legible and portable: a two-typeface system where weight does the hierarchy work rather than a large size ladder, a token-minimal theme driven by five or six CSS custom properties as RGB triples, a clean three-pane shell, and a tri-state light/dark/system toggle. Its 2026 repositioning leans agent-native, serving every page as raw Markdown, auto-generating `llms.txt`, and exposing a per-page "Copy page" affordance. Its known weakness, deep sidebar nesting with no depth indicator, is itself a documented lesson worth stealing the fix for.

**Starlight.** Starlight is the anatomy Topo ports, and it is loved for shipping a complete, accessible docs skeleton that a project either uses as-is or overrides surgically through a 29-slot component registry without forking. Its search is zero-config Pagefind indexing the built static HTML, so there is no account and no crawler. Its sidebar works in two modes, hand-authored and directory-autogenerated, and the two compose. Its accessibility posture is a genuine draw: a skip link as the literal first body element, semantic landmarks, and heading-anchor labels that stay screen-reader-navigable while being excluded from the search index. Its theming is a clean `--sl-*` token namespace layered through CSS cascade layers, so custom CSS wins over defaults without `!important`, and its zero-JS-by-default output typically ships under 50KB per page.

## 2. Devices to absorb

| Device | Source | What it does | Why Topo wants it | Cost |
|---|---|---|---|---|
| Zero-config static local search | Starlight (Pagefind) + VitePress (minisearch) + Fumadocs (Orama) all converge | Indexes built HTML client-side, no account, no external service, modal on `Cmd/Ctrl+K` | Already Starlight's default; the whole category agrees this is table stakes and the right floor for small/medium docs | Cheap (keep Pagefind) |
| Path-scoped / root-folder sidebar tabs | VitePress (path-prefix trees), Fumadocs (root-folder tabs) | Surfaces a tab switcher so guide vs reference vs explanation carry independent nav trees | cairn docs are Diataxis-shaped (reference/guides/explanation/tutorial); readers orient by arm, not one flat tree | Moderate |
| Depth-indicated sidebar nesting | Mintlify (as the fix for its own documented gap) | Indent plus a connector line or tint so deep nesting reads its level at a glance | Cheap correction of a known best-in-class weakness; a clear differentiator | Cheap |
| Intersection-Observer TOC with auto-scroll | Fumadocs | Active heading highlights and the TOC panel scrolls to keep the active entry in view | Starlight already tracks the active heading; the auto-scroll-into-view refinement is the polish delta on long reference pages | Cheap |
| Automatic prev/next footer | Starlight + VitePress (both derive from sidebar order) | Previous/Next links inferred from resolved sidebar order, per-page overridable | Turns a linear guide into a guided path with no hand-maintained "next steps" links that rot on reorder | Cheap (Starlight ships it) |
| `code-group` tabs with synced, persisted selection | VitePress | Groups related fences into tabs (npm/pnpm, JS/TS), a `name` param syncs the reader's choice across every group on the page via localStorage | The single most repetitive docs-authoring pattern; syncing means the reader chooses once | Moderate |
| Live-file code imports | VitePress (`<<< @/path`) | Pulls real source into a fenced block instead of a hand-copied snippet, with region markers | Keeps documented cairn code from silently drifting out of sync with the actual source | Moderate |
| Card-grid pattern for index/quickstart pages | Mintlify (`CardGroup`) + Fumadocs (Cards) | Breaks linear prose into a composed grid on landing and index pages | Front doors of each docs arm read as composed, not a wall of links | Cheap |
| Two-typeface, weight-driven hierarchy | Mintlify | One sans for body and headings, one mono for code, weight (not many sizes) carries hierarchy | Aligns with the family identity discipline; the port wears cairn's type, not Inter, but adopts the weight-over-size method | Cheap (CSS) |
| Token-minimal theming as the family skin | Mintlify method over Starlight's `--sl-*` structure | A small set of custom properties as the source for both light and dark and alpha variants | This is where the family identity lands on Starlight's anatomy; Warm Stone tokens map onto the `--sl-*` slots | Moderate (the identity work) |
| Tri-state theme toggle (light/dark/system) | Mintlify + Starlight (`ThemeProvider` already reads `auto`) | Three-state selector, pre-paint inline script to avoid flash, persisted to localStorage | Starlight's provider already supports `auto`; expose it as an explicit three-state control | Cheap |
| Serve each page as raw Markdown + `llms.txt` + "Copy page" | Mintlify | Parallel `.md` URL per page, generated `llms.txt`/`llms-full.txt`, a copy-page affordance | Cheap, high-signal 2026 docs table stakes; trivial as static SvelteKit routes | Cheap |
| Config-first site definition | Mintlify (`docs.json`) + Starlight (`starlight()` config) | One file drives nav, colors, fonts, arms so a maintainer changes the site without touching components | Matches cairn's seam philosophy and its extending-developer lens | Moderate |
| Component-override registry survives updates | Starlight (29 slots) + Fumadocs (component-fork CLI) | A documented override map swaps individual components without forking the theme | Directly serves cairn's contract that a developer's work survives engine updates | Moderate (it is the port itself) |

## 3. Devices rejected

- **Interactive OpenAPI "try it" playground** (Mintlify, Fumadocs). Real engineering against a spec runtime; cairn docs do not document a live API, out of scope until they do.
- **Hosted AI assistant / conversational search panel** (Mintlify). A SaaS backend with metered AI-credit pricing; not self-hostable, not lean, not free.
- **Algolia DocSearch as the default search** (VitePress, Starlight swap-in). Adds an account and a crawler; keep it only as a documented escape hatch, never the default over Pagefind.
- **Markdown-as-Vue-SFC / RSC server-first authoring** (VitePress, Fumadocs). Framework-coupled authoring models; cairn docs are static Markdown-in-git and should stay renderer-agnostic.
- **Twoslash inline type-checking in code samples** (Fumadocs). Heavy TS-toolchain integration for marginal payoff on a lean docs shell; revisit only if reference pages demand it.
- **Auto Type Table from TypeScript types** (Fumadocs). Same reasoning; a niche generator that couples docs to a TS build.
- **Native versioned-docs system** (neither VitePress nor Fumadocs ships it well; Docusaurus does). Not needed at cairn's scale; folder or build convention if it ever is.
- **Blog features: RSS, tags, archives** (weak even in VitePress). Topo is a reference-docs shell, not a blogging platform.
- **Component-eject CLI as a shipped feature** (Fumadocs shadcn pattern). Absorb the override philosophy, not a separate CLI surface; Starlight's slot registry already covers it.
- **Hosted analytics** (Mintlify). A build-vs-buy the site owner decides, not a shell feature.
- **1600px content container** (Fumadocs v16). Runs against the restrained centered-column reading measure Topo wants from VitePress and Mintlify.

## 4. Starlight anatomy checklist (must stay recognizable)

The port keeps these structural habits legible under the family skin.

- **Layout regions in order**: skip-link, header (site title/logo, search, theme select, social), left sidebar column (desktop) / off-canvas panel (mobile), centered main content column, right TOC column (desktop) / sticky dropdown (mobile), footer (last-updated, edit-link, prev/next pagination).
- **Sidebar dual mode**: hand-authored structure and directory-autogenerated groups that compose, with per-page order override, collapsible groups, and badges.
- **Search as a header-button modal** on `Cmd/Ctrl+K`, Pagefind against the built static HTML, per-page and partial-page index exclusion.
- **On-page TOC** from `h2`/`h3` by default, configurable depth, active-heading tracking, and a separate mobile presentation of the same heading tree.
- **Prev/next pagination** derived from resolved sidebar order, per-page overridable.
- **Three independent responsive collapses**: left nav to off-canvas, right TOC to sticky dropdown, header-only controls reflowing into the mobile menu footer rather than disappearing.
- **Accessibility posture**: skip-link as the literal first body element, semantic `header`/`nav`/`main`/`footer` landmarks, heading-anchor accessible labels excluded from the search index.
- **Theming through a token namespace** layered via CSS cascade layers (custom CSS wins without `!important`), `data-theme` on a bare wrapper per the admin design rule, pre-paint theme script to avoid flash.
- **Zero-JS-by-default weight budget**: the port keeps Starlight's light-payload character rather than adopting a heavier SPA model.

## 5. Open design questions for mockups

- **Sidebar-arm switcher form.** Tabs at the top of the sidebar (Fumadocs) or path-scoped trees with a section header (VitePress)? Does cairn's four-arm Diataxis map to four tabs, or do reference and guides dominate while explanation and tutorial live inside?
- **Family identity intensity.** How far does Warm Stone push before the port stops reading as Starlight underneath? Where is the line between "recognizable anatomy" and "glance-indistinguishable" for type, sidebar density, and accent use?
- **Typeface decision.** Does Topo wear the admin's Bricolage/Figtree pairing, or does a docs shell want a different reading face? What is the mono for code blocks?
- **TOC placement at the extremes.** Right rail at 1440 and 2560 is clear; what happens to the three-pane shell at 2560 (max content measure vs. filling the width), and how do left nav and right TOC coexist at 768 before both collapse?
- **Depth indicator style.** Indent-plus-connector line, tint ramp, or both, and how deep does nesting go before it should be flattened instead of indicated?
- **`llms.txt` and per-page Markdown scope.** Ship for every page from day one, or reference arm only? Is the "Copy page" affordance in the header, the footer, or beside the title?
- **Code-group breadth.** Which variant axes does cairn actually need (package manager, JS/TS), and is synced selection worth the localStorage surface for a docs set this size?
- **Card-grid usage.** Which pages get the grid (each arm's front door, the tutorial index) and which stay linear?
- **320 and 2560 composition.** The standard requires the extremes be composed, not merely unbroken. What does the header collapse to at 320, and does the 2560 view add anything (wider TOC, persistent breadcrumbs) or just center?
- **Search backend at family scale.** Pagefind default is settled; do any family docs properties reach the size where the Algolia escape hatch matters, and is that hatch documented now or later?
