Media management — direction A: insert-first (inline)
The author lives in the editor. Media arrives in place by drop, paste, or slash, through a compact at-cursor picker popover, never a full-screen modal. The full library is a secondary management view in the nav, for cleanup and audit. One insert entry point routes by intent to the three lifecycles: stored files (R2-backed, the build target), referenced media (paste a URL, oEmbed, a directive, nothing stored), and design-system tokens (icons, picked never uploaded). Shared scenario: a posts-and-pages ski site, a 14-asset library with clean and messy names. Light theme is canonical; one dark panel proves the tokens carry.
1 · The primary surface — the desk, insert chooser open at the caret
The everyday state. The author typed / on an empty line (or clicked the toolbar insert glyph, highlighted), and the chooser opened anchored to the caret, not over the page. Three routes by intent. The live preview tab stays one click away; the manuscript never leaves the screen. This is the door for all three lifecycles: "A photo or file" is the stored library, "Video or embed" pastes a URL, "An icon" opens the token picker. Drop and paste reach the same place without the menu.

4 · The differentiator moments — no dead interval, failure + retry, dedup, safe delete
These are how cairn beats the field. 4a the upload has no dead interval: an optimistic placeholder lands at the caret the instant you drop, with determinate progress, then swaps to the committed reference (the button runs it). 4b a failed upload shows an inline error with a retry, never a silent drop. 4c dedup on upload: identical bytes resolve to the existing asset, made visible. 4d "where is this used" plus safe delete: an in-use asset's delete is hard-confirmed and lists the entries. Reuse and replace-in-place are panel 2 and panel 6.

6 · Replace-in-place — swap the bytes, keep every reference
The single most-praised capability in the field is Webflow's replace-an-asset-and-it-updates- everywhere. Because the reference is a stable media: handle, replacing the bytes updates every placement at once. New bytes mean a new hash, so the cache busts on its own; the reference in the content never changes.

10 · Accessibility — focus order, keyboard map, screen-reader narration

Designed-in accessibility for the insert flow

The primary flow is keyboard- and screen-reader-reachable end to end: open the chooser, route to the library, pick or upload, then capture alt. The mouse path (drop, paste) and the keyboard path (slash, the insert glyph, Ctrl+Shift+M) converge on the same surfaces, so nothing is mouse-only. Contrast holds the design-system floors; muted text is never stacked on opacity, and every state has a non-color cue (a check glyph, a labeled badge, an icon).

Focus order — the at-cursor flow

  1. Open. The insert glyph carries aria-haspopup="menu" and aria-expanded. Slash and Ctrl+Shift+M open the same chooser.
  2. Chooser. A role="menu"; arrow keys move between the three routes, Enter activates, Esc closes and returns the caret to the document where it was.
  3. Library popover. Focus lands on the search field. Tab order: search → type-filter chips → grid → asset detail link → pager → Upload → Insert → Close. The grid is one composite widget (a roving tabindex; the arrow keys move between tiles, so the grid is one tab stop, not 14).
  4. Capture card. On upload or paste, focus moves to the file-name field, then alt, then the Decorative checkbox, then Cancel and Insert. Insert is disabled until alt is filled or Decorative is checked, and the disabled button names why.
  5. Return. On insert or close, focus returns to the editor at the caret, and the new reference is selected so the next keystroke is in context.

The popover is a non-modal anchored surface, so focus is not trapped: Esc or a click outside dismisses it and restores the caret. The capture card and the safe-delete dialog are modal (role="dialog" / role="alertdialog"), so they trap focus and restore it on close, the design-system dialog recipe.

Screen-reader narration — the primary flow

"Insert media, menu button, collapsed." Activate.
"Insert media, menu. A photo or file, menu item, one of three." Down arrow twice, Enter on it.
"Your library, dialog. Search name, alt, or caption, search field." Type "shoes".
"Showing 1 of 14" (the count is an aria-live="polite" region). Arrow into the grid.
"blue-running-shoes.jpg, 2400 by 1600, 612 kilobytes, used in 3 entries, 1 of 1." Enter.
"Inserted blue-running-shoes.jpg. Editing." Focus is back in the manuscript.

Each tile's accessible name carries the full picture, name, dimensions, size, and usage, so a non-sighted author hears what a sighted one sees on the detail rail. The upload's progress is an aria-live region that announces start, a coarse percentage, and the committed result, so the no-dead-interval promise reaches a screen reader too. A failure is a role="alert". Alt capture is a labeled required field with the Decorative escape hatch as a real checkbox, so the one accessibility-critical field is never a silent skip.
