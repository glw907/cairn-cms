Media Pass B — Replace and Alt propagation (in-context direction)

Two new operations on the Media Library detail slide-over at /admin/media. Replace swaps an asset for a freshly uploaded file and repoints every published reference to the new content-addressed hash in one commit. Alt propagation pushes the asset's default alt to placements that lack it, with an opt-in to also overwrite custom alt. Both rewrite N published entries at once, so this direction keeps the whole flow inside the existing non-modal slide-over: each operation expands an inline preview region in place (impact count, per-entry old to new, the report-only branch-delta) and confirms in place. The asset stays in view; no modal, no route change. The danger family is spent only where earned: the fail-closed verification block and the overwrite opt-in.

Light: the slide-over at rest, with the Replace and Push-alt entry points

The resting panel is the established detail slide-over: the preview, the name and media: reference, the default-alt editor, where-used grouped published-then-branch, and metadata. The new operations are two quiet controls in the actions block: Replace and Push alt. They sit beside the existing Delete, all text-weight, so the panel stays calm until the author opens one. The grid behind stays live and in the a11y tree (no scrim), lightly dimmed to focus the panel.

Light: Push alt to placements (the three buckets)

Push alt expands its own region. It groups affected placements into three buckets. Will fill are empty-alt placements that gain the default (the safe, default action). Already described are placements with custom alt, skipped unless the author ticks the opt-in to overwrite them; the checkbox and the count live in the danger family because overwriting an editor's words is the destructive move. Decorative heroes are skipped per the persisted flag. The footer count tracks the opt-in: it reads the fill count alone, then fill plus custom when the box is ticked.

The body-image caveat. A body image cannot record a decorative choice, because markdown's alt has no slot for it. So an empty-alt body image always reads as a fill gap, never as decorative-skipped; only a frontmatter hero can be decorative. The Will-fill bucket carries one plain line to that effect when any of its rows is a body image, so the author is not surprised that a deliberately-blank body image shows up to be filled.

Light: Replace fails closed (usage could not be verified)

A replace must rewrite every reference, so it fails closed if usage cannot be fully verified: an open branch that failed to read, or a stale usage index. This is the one place the danger family carries the whole region. It states the reason plainly, names what could not be read, and offers the only safe next step (re-run the scan), with no apply button at all. The author cannot half-replace published content.
