Media Pass B — Replace and Alt-propagation — rev.2 (consolidated)

Two new operations on the existing Media Library detail slide-over at /admin/media. Replace uploads a new file for an existing asset. cairn is content-addressed, so the new file has a new hash and every reference in published content is repointed to it in one commit to main. Alt propagation pushes the asset's default alt to placements that lack it, with an opt-in to also overwrite placements that carry custom alt. Both rewrite N published entries at once, so the UI's whole job is to make the impact legible and confirmable before applying. Each operation opens its own dedicated review modal (~38rem, single column, the modal-box is the scroll container) from the resting non-modal slide-over. Replace is role="alertdialog" with the danger family and a typed-slug confirm; alt fill is role="dialog" with the primary register and a single bucket-level opt-in. Read the rev.2 decisions in the source comment at the top of this file.

1 · Light: the slide-over at rest, with the Replace and Push-alt entry points

The resting panel is the shipped detail slide-over. The actions block gains two quiet text-weight controls beside the existing Delete: Replace image and Push alt to placements. They are levelled controls, lighter than a bordered row, and neither acts until its review modal is read and confirmed. Delete keeps the shipped danger-bordered control. The grid behind stays live and in the a11y tree (the non-modal region recipe); a modal scrim only appears once a review opens.

5 · Light: Push alt to placements — the three buckets (opt-in unchecked)

Alt propagation is the everyday register: role="dialog", no typed gate. The work-tuned title is "Fill alt on 5 placements". The alt being pushed is shown once so the author confirms the text. Will fill (empty alt) is always applied and shows every row's honest (no alt) → default-alt. Has custom alt is opt-in through one real native checkbox; before it is checked, each row's existing custom alt is shown plain and marked "kept", so the author reads what is at risk first. The opt-in band is styled in the danger family, because overwriting an editor's words is the destructive choice. Decorative hero, skipped is reported and never an input. The body-vs-hero caveat sits beside will-fill.

role="dialog" (NOT alertdialog) · the opt-in is a REAL native input type="checkbox" class="checkbox checkbox-sm" inside its label (peer/:checked-driven, in the a11y tree, keyboard-operable) · checking it is the only state that changes the total
