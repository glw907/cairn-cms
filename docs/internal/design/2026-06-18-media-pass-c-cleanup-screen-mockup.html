Media Pass C — the destructive cluster — direction: the deliberate Clean up ritual

Pass C adds the destructive media operations the Library deferred: multi-select bulk delete on the browse Library, and orphan collection (the destructive half of the read-only reconcile) on a dedicated Clean up surface, a second tab on /admin/media. This direction reads orphan collection as a deliberate hygiene ritual that deserves its own focused, itemized surface, distinct from browsing. Ad-hoc bulk delete stays in the browse Library (panels 1–3); the ritual sweep lives on Clean up (panels 4–7). The whole cluster honors one safety floor: the displayed usage index is non-strict, so it is for selection only; every destructive item re-verifies with its own strict read and fails closed. Read the load-bearing decisions in the source comment at the top of this file.

1 · Light: the browse Library with multi-select armed (the sticky action bar)

Ad-hoc bulk delete lives where you browse. A Select control arms multi-select; each tile gains a real native checkbox in its corner. The grid is the shipped roving listbox, now aria-multiselectable: focus and selection are decoupled, Space toggles the active tile, Shift+Arrow extends a range, Ctrl/Cmd+A selects all in view, Escape clears. A sticky bar names the selection count and its explicit scope ("the 6 in this view"), offers select-all-matching as a separate convenience, and carries the one bulk action. The selection is a convenience over the displayed index; the safety gate is per item at execution, said at the confirm.

role="listbox" aria-multiselectable="true" · roving tabindex (one tabstop) · Space toggles selection on the active option (decoupled from focus) · Shift+Arrow extends a range · Ctrl/Cmd+A selects all in view · Escape clears the selection · each tile checkbox is a real native input in the a11y tree the sticky bar's "N selected" is a polite sr-only role=status live region too, so a screen-reader hears the count change as tiles toggle

2 · Light: bulk delete — the per-item-gated confirm (skip and report)

The bulk delete confirm is the single safe-delete gate applied per item, never weakened. It splits the selection honestly by a fresh strict re-read: the items it can delete now (no references found) and the items it will skip because they are still in use, each naming where. In-use items are never bulk-force-deleted; forcing a delete stays the deliberate single-item typed-slug path. Because every removal is one revertible commit, the confirm is plain (it names the count), not a typed gate. The footer button counts only what will actually be deleted.

role="alertdialog" · showModal() (native focus trap + Escape) · NO method="dialog" backdrop (no light dismiss on a destructive surface) · initial focus is Cancel · the split is recomputed from a fresh strict per-item read, NOT the loaded display index

4 · Light: the Clean up surface — at rest, then the scanned two-section dry-run

Clean up is the ritual's own surface, a second tab on /admin/media. At rest it is a calm detect screen: one Scan for orphans button that runs the reconcile plus the cross-branch usage scan. After the scan it shows the two-section dry-run: Orphaned files (Direction A: R2 bytes with no record) and Unreferenced images (Direction B: records nothing uses). Each section states what it is and what cairn cannot see; neither has a one-click "delete all". This panel shows the rest state (left) and the scanned overview (right, sections collapsed to their heads; the full itemized wells are panels 5 and 6).

the scan runs the strict cross-branch read; a degraded branch read downgrades the whole scan to "could not finish" rather than under-reporting orphans · the Clean up tab's count badge reflects the last scan

5 · Light: Clean up — Direction A (orphaned files), the itemized dry-run

Direction A is the lower-friction tier. These are R2 bytes with no library record, so no media: token resolves to them: "no reference" is close to proof. The dry-run is mandatory and itemized, each row a structured fact (the storage key, the size). Per-row checkboxes plus a select-all-in-section convenience let the operator choose what to purge. Because there is no record, there is nothing in git to revert: this purges the bytes for good, said plainly and kept distinct from the reversible removals elsewhere. The one residual risk (a raw-HTML page that hardcodes the delivery URL) is named.

each row checkbox is a real native input · the section header checkbox is the select-all (indeterminate when a subset is checked) · "Purge N files" opens the typed confirm (panel 7) because the byte delete is irreversible

6 · Light: Clean up — Direction B (unreferenced images), the strict dry-run

Direction B is the less-safe case. These are real library records that no entry references, but they ride the same raw-HTML blind spot as a single delete, so the section restates the caveat inline and each row's where-used reads the honest "found in 0". It inherits the full strict fail-closed gate: the "Remove" trigger opens a typed confirm, and each item is re-verified per item at execution against a fresh strict read, never the displayed scan. Because each removal is a git commit, the records and bytes can be brought back; this is the reversible side of the surface, kept distinct from Direction A's irreversible byte purge.

each item is re-verified with its own strict cross-branch read at execution; an item that cannot be verified is skipped and reported, never removed · "found in 0" is the displayed scan, the selection convenience, not the gate

7 · Light: the two batch confirms — A is typed (irreversible purge), B is plain (revertible)

The two directions earn two confirm shapes, the load-bearing reversibility distinction. Direction A (left) purges R2 bytes with no git record, so it is irreversible: it carries the danger register, an explicit "cannot be undone" block, and a typed confirm (type purge). Direction B (right) removes git-tracked records and bytes in one revertible commit, so it earns a plain confirm that names the count and states the git-revert reassurance, plus the standard fail-closed promise that any item found in use on the recheck is skipped. Same surface, two clearly different weights.

A: role="alertdialog", typed confirm gates the apply (the one legitimate disabled submit) · B: role="alertdialog", plain confirm naming the count, no typed gate (reversible) · both: showModal(), no light dismiss, initial focus on Cancel, focus restores to the section's trigger on close

8 · Light: Clean up — the scan could not finish (fail closed)

The whole surface rests on a strict cross-branch read. If a branch will not load, the scan does not under-report orphans by quietly skipping it: it refuses to present a dry-run and names the branch it could not read, exactly the safe-delete fail-closed posture applied to the sweep. There is no collect action, not even a disabled one. A quiet Try the scan again re-runs it. Nothing was changed.

the banner is role="status" on first load (no action attempted) · role="alert" would be used only after a collect was attempted and failed · no collect button present anywhere on a failed scan
