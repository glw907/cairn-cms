Media Pass C — rev.2 consolidated: the destructive cluster

The approved design target. The base is the in-library direction: the destructive cluster extends the shipped Media Library at /admin/media in place, with no new route and no tab. Three populations are kept distinct (the critics' top finding). Unreferenced assets (the renamed No references found triage facet) are reversible git deletes, handled by multi-select bulk delete in the Library. Orphaned bytes (stored files with no library record) are an irreversible purge, handled by a separate on-demand scan opened from a quiet Find orphaned files button, never the triage. Broken references (a record whose bytes are gone) are a read-only data-integrity readout in that same scan. The reversible and irreversible paths are never reachable from one button or one modal. The safety floor: one shared strict cross-branch index built per batch (fail closed), rechecked per item at execution, never the displayed count. cairn is not WYSIWYG: every before/after is a structured fact. Read the full decisions in the source comment at the top of this file.

1 · Light: the Library, multi-selectable, nothing selected (the renamed facet, the scan entry)

The shipped grid, now multi-selectable. Each tile carries a real native checkbox at top-left, decoupled from the roving focus: arrows move focus, Space toggles selection. The triage radiogroup stays asset filters only, three peers: the shipped All / Needs alt, then the renamed No references found (was "Unused"). Orphaned bytes are not a triage facet; they live behind the quiet Find orphaned files button beside Upload, because they are not library assets and detecting them is a heavier scan. The top-right usage chip is renamed too: "Unused" becomes No refs. With nothing selected there is no action bar; the surface reads as it does today.

3 · Light: bulk delete confirm — the dry-run is the selection, with the skip-and-report split

The confirm dialog is the dry-run: it lists exactly what will be removed and what will be skipped, so there is no separate preview step. The scope strip names the count and discloses the safety floor, and it is careful about timing (the critics' review-time-vs-execution-time fix): it says the shown usage came from a quick read and each item is checked again when it deletes, not that the gate has already passed. Will be deleted lists the no-reference items, each with its slug and a "no references" tag. Will be skipped reports the one item still in use, with its where-used, because a bulk delete never force-removes an in-use asset; it points to the single-item path for a deliberate forced delete. This is a reversible removal, so the confirm is plain (no typed gate): the button reads Delete 2, skip 1 in the danger-outline, and the git-revert reassurance sits at the foot.

4 · Light: bulk delete — announced progress, then the itemized summary

5 · Light: the No references found facet (unreferenced assets) — the reversible removal, plain confirm

6 · Light: Find orphaned files — the scan result, two sections (orphaned bytes + broken references)

The on-demand scan opens as a modal over the media view, not a standing tab. It runs an R2-list plus a reconcile pass, then shows two sections, and it keeps the two reconcile populations distinct. Orphaned files are R2 bytes with no library record: there is no tile, so each row is a structured byte fact (the storage key, the size, a last-seen) with a checkerboard mat standing for record-not-picture, so an orphaned byte never borrows a misleading image. The section has an indeterminate header checkbox as select-all, real per-row checkboxes, and the verb Purge, because removing bytes is irreversible. Broken references is a separate read-only data-integrity readout: records whose bytes are gone. It is not a deletion target; it says what is wrong and what to do (re-upload or remove the reference).

the orphaned-byte rows are a separate aria-multiselectable list role="listbox" aria-label="Orphaned files" · the header checkbox is the select-all (indeterminate on a subset) · byte-orphans use their OWN selection Set of R2 keys, never the asset-hash Set · the broken-references section has NO checkboxes (read-only) DETECTION COST: this scan is far heavier than the loaded index (an R2 list + reconcile), which is why it is on-demand, not an instant filter · it builds its own strict read and fails closed (panel 8)

7 · Light: purge orphaned files — the irreversible confirm with the typed gate

The byte-purge confirm is the one place in the cluster that reserves a typed confirm, because it is the one irreversible action, and it is reached only from the scan surface, never from the Library's selection bar. It lists the keys and bytes to remove (the dry-run), states plainly that this cannot be undone and that there is no git history for raw bytes, restates the blind-spot caveat, and gates the purge behind typing the file count (the value is tied to the actual set, the shipped safe-delete idiom, not a reflexive word). The title and the action stay verbally distinct from a normal delete: Purge, never "Delete". The badge and the button carry the solid danger fill (the one fill the destructive register owns; warning-ink is never a fill).

role="alertdialog" · the typed gate is reserved for THIS path only (irreversible) · the input is type="text" matching the file-count value · the solid danger button stays disabled until the typed value matches

8 · Light: fail closed — the bulk delete on hold (left) and the orphan scan that could not finish (right)

The safety floor in action, on both paths. Left: when cairn builds the shared strict index for a bulk delete and a branch read fails, the op holds before touching anything, names what read cleanly and the specific unreadable branch, and offers a quiet Check usage again. There is no delete button, not even a disabled one. Right: the orphan scan rests on the same strict read; if a branch will not load, it does not under-report orphans by quietly skipping it. It refuses to present a dry-run, names the branch it could not read, and shows no collect action. Both are the shipped fail-closed posture, one applied to the bulk gate and one to the detection scan.

both banners on first appearance are role="status" (polite, no action attempted) · role="alert" is reserved for a failure surfaced AFTER Delete or Purge was pressed · no destructive button present on either · the selection / scan input is held, nothing changed
