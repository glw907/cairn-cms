Tidy settings: the final polished mockup

This is the polished settings surface for tidy, the light, voice-preserving copy-edit from Claude. It is the winning grouped-toggle-list direction, hardened against the synthesis brief and rebuilt on the corrected convention set. The chosen editor surface is the review-mode mockup, so this screen shares its diff language: the on-state examples here use the same strike-and-add runs the editor later confirms in review.

Two clearly separated sections carry the work. Fixes are the objective errors that default on, settled and rarely touched. Style conventions are the choices that default off, an opt-in menu; turning one on reveals a pick-one variant chooser. Above both sits a generated summary line in a live region that states the whole config in one read. The resting state is the safe default: Fixes on, Style off, every variant collapsed, zero decisions asked.

The two-tier split is load-bearing. The developer turns tidy on, supplies the API key, and picks the model at deploy time; the editor governs the conventions here. All three deploy-time facts show read-only at the top. When the developer has not enabled tidy, this whole surface is replaced by a short, honest note (state 4), never a teasing disabled form.

State 1 · Editor tier, enabled, at rest (light)
The everyday state. Fixes are all on (they are errors, not choices); Style conventions are all off (cairn leaves the author's choice until someone declares it); Advanced is collapsed. The developer tier shows read-only at the top, including which model runs. The summary line states the whole config. Nothing here demands a decision; the editor can leave this screen exactly as it loads.
a11y. Each on/off control is the shipped check-and-tint aria-pressed button, so state lives in the control and the check glyph, never in color (WCAG 1.4.1). The summary line and both section counts sit in an always-present role="status" / aria-live="polite" region (the EditPage needs-alt and MediaPicker-count precedent), so a toggle announces the new total. The tiny diff examples are aria-hidden; a screen reader hears "Spelling and typos, on", not the struck runs.
engine. The convention set is committed per-site YAML, edited through the GitHub-App commit pipeline, so it is diffable and shared across editors. The system prompt is built from the enabled conventions only; an off convention contributes no rule line, so the model is never told to touch it. The model and key are deploy-time facts read from the Worker config, surfaced here read-only.

State 2 · Two style conventions on, variant choosers open (light)
The active state. Turning a style convention on reveals a pick-one chooser for its variants; the example under it shows the exact effect of the chosen variant, and the generic row example is suppressed so the row never shows two examples. The summary line and the section count both update to match. Every chooser is the shipped radiogroup with roving tabindex and a check glyph, never an aria-pressed toggle.
a11y. Each variant chooser is the shipped pick-one recipe (CairnMediaLibrary's triage): role="radiogroup" wrapping role="radio" segments with aria-checked; the selected radio is the only tabstop (tabindex="0", the rest -1); ArrowLeft/Right and Home/End move and select with wraparound; the check glyph is the non-color cue (WCAG 1.4.1). The chooser mounts only when the row is on, so a screen reader never meets an orphan control. The summary line and counts re-read through their live regions.

State 4 · The visibility gate · tidy not enabled (light)
What the editor sees when the developer has not turned tidy on. The convention list is gone, not disabled. In its place is one honest note: tidy is a developer setup, here is what it needs, and here is who does it. The literal config tokens sit in a marked developer aside, not the editor line. Spellcheck still works because it is local, so the screen reassures rather than reading as broken.
a11y. The gate is the empty-state recipe, not a form: no disabled controls live in the tab order, so a keyboard or screen-reader user is never offered config they cannot use. The whole note is a labelled role="region" landmark; the checklist is read-only text, the one link goes to the docs.
engine. The editor tier renders only when tidy is enabled AND the API key secret is present (the brief's MK-4 gate, extended to the whole settings section). When either is missing, the route serves this state instead of the convention list.
