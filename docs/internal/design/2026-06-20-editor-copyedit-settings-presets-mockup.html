Tidy settings — Direction "presets"

The tidy convention config, fronted by presets. One click sets a sensible bundle: Typos only (the safe default, objective fixes alone), Light house style, a Chicago-leaning set, an AP-leaning set, and Custom. Under the presets, an Advanced disclosure opens the full per-convention list, each style convention carrying its variant choice. A non-expert reaches a good config from one choice; a power user fine-tunes every line.

The two risks the brief names are handled head-on. A plain-language summary under the preset always states, in words, what tidy will and will not change, so a preset never hides what is on. And the selection only ever shows a named preset when the config matches it exactly; the moment you change one convention it becomes Custom, so the chip never lies and Custom stays an honest label, not a sixth bundle.

Two tiers with a visibility gate. The developer-tier facts (tidy is enabled, the API key, the model) are shown read-only at the top. The editor-tier config is rendered only when the developer has enabled tidy; when they have not, the section is replaced by a short "ask your developer" note, never a teasing disabled form. The states are stacked below so the design reads fully.

State A · The editor-tier surface, tidy enabled (light)
The everyday settings screen. Up top, a read-only strip states the developer-tier facts the editor cannot change. Below it, spellcheck (local, on by default), then the tidy convention card. The preset is Typos only, the safe default, so the summary states tidy fixes the objective errors and leaves every style choice alone. Advanced is closed because the preset is enough on its own.
engine — the developer tier (the tidy.enabled switch and the Anthropic API key Worker secret) is set at deploy time, not in this UI; the strip reads it back and renders it locked. The editor tier (the convention config) renders only because both are satisfied.
