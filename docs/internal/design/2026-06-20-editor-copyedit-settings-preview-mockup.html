Tidy settings — Direction "preview": example-first preview cards

Tidy is a light, voice-preserving copy-edit from Claude. This is where an editor governs it. The governing idea of this direction: every convention is a card whose main content is a real before and after example. The editor learns each one by seeing it, not by reading grammar terminology, and the toggle sits on the card so the effect is visible before it is turned on.

The objective fixes are on by default and read as a tight, scannable list. The style conventions are off by default; each one shows what it would do as a single muted line and opens its variant chooser only when the editor turns it on, so a fifteen-convention surface still scans. Two tiers are separated: the developer sets up tidy and holds the key (shown read-only), and the editor configures the conventions. When the developer has not enabled tidy, the convention section is absent, replaced by a short note (screen 4).

Screen 1 · The full surface at rest, safe defaults (light)
The resting state, which is also the safe default. The developer-tier card at the top is read-only status. Below it, the six objective fixes are on (each a card with its example), and the nine style conventions are off (each a compact card showing what it would do, chooser collapsed). The editor changes nothing here and the settings are already correct.
a11y — each convention toggle is a real <input type="checkbox"> with an accessible name from the card heading; the on/off state lives in the control, never in color alone. The two section counts are quiet text, and the developer card is plain read-only content (no disabled form controls to confuse a screen reader).
engine — this whole editor-tier section renders ONLY because the developer enabled tidy and the key is present (screen 4 is the other branch). The per-convention values persist to a git-committed per-site YAML file, edited through the GitHub-App commit pipeline, so the config is diffable and shared across editors. The system prompt is built from the ENABLED conventions only.

Screen 2 · A style convention turned on, variant open (light)
What happens when an editor turns on a style convention. The variant chooser opens and the card's example re-renders to the chosen variant, so the editor sees their own setting, not a generic one. Dash style adds a small extra toggle for the double-hyphen rule. Number style is a three-way choice. The card border lifts so an on convention is easy to find again in the grid.
a11y — the variant chooser is the segmented check-and-tint recipe: a role="radiogroup" of segments with aria-checked and a roving tabindex; the active segment shows a check glyph so the choice is never carried by color alone. The double-hyphen sub-toggle is a labelled checkbox. The example updates are presentational and not announced on every keystroke.
engine — only the enabled variant emits a rule line into the system prompt (poplar's BuildPrompt model). "Spaced em-dash" and "unspaced em-dash" are mutually exclusive variants of one convention, so exactly one line is emitted, never a contradiction. Regional spelling is never normalized by any of these; it stays with the author and the spellcheck dictionary.

Screen 4 · The visibility gate — tidy not enabled by the developer (light)
The honest empty state. When the developer has not turned tidy on, the editor does not see the convention cards at all. They see a short note that says what tidy is and what a developer needs to do. No disabled toggles, no preview of a feature the editor cannot use. This is the truthful half of the two-tier gate.
a11y — this is plain content, not a disabled form, so a screen reader reads an explanation rather than a list of controls the user cannot operate. The note names the concrete developer steps so the ask is actionable, never a dead end.
engine — the editor-tier section is conditionally rendered on tidy.enabled && keyPresent, both derived server-side from the Worker secret and the config. When the gate is closed the convention list is never sent to the browser, so there is nothing to reveal in dev tools either.

Tidy uses Claude, so it sends your draft to the Anthropic API and costs a little per use. That makes it a setup step for whoever runs your site. Once it is on, the writing conventions appear here for you to choose.

Ask your developer to:

  1. Turn on tidy in the site config (tidy.enabled).
  2. Add an Anthropic API key as a Worker secret.
