# Admin re-expression Phase 1: the vocabulary-screen pilot, on the frozen idiom

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking. Task 1 is a main-loop, mockup-first design stage, not a code task.

**Goal:** Build the vocabulary admin screen — the idiomatic-re-expression initiative's proving pilot — natively on the Phase 0 frozen role interface and walled Tier-2 foundation; finish tag-management Plan 3's showcase filter, e2e, and seed/orphan flow on that idiom; extend the per-phase visual baseline to the pilot screen; then cut the held tag-management release.

**Architecture:** The vocabulary screen is an **engine** admin screen, a near-twin of `CairnTidySettings.svelte`: a `VocabularyAdmin.svelte` rendered in the `CairnAdmin.svelte` view switch, linked from the `CairnAdminShell` sidebar, posting a hidden-JSON working copy through the already-landed `vocabularyLoad`/`vocabularySave` route pair (`9ba8791`). It is built in native DaisyUI 5.6 / Tailwind 4 primitives and the frozen `text-muted` / `text-subtle` role utilities, so it adds zero bespoke custom surface: it is the proof that a screen built on the documented vocabulary reads idiomatic and passes the gate on day one. The showcase filter is template-only, a client-side narrowing of the rendered archive over `ContentSummary.tags`.

**Tech Stack:** TypeScript, Svelte 5 (runes), DaisyUI 5.6, Tailwind 4, the Warm Stone design system and its frozen role layer; Vitest (chromium component tests; GithubDouble route tests, already landed); Playwright (the showcase e2e and the visual baseline).

## Global Constraints

- **Two specs are canonical.** The idiom is governed by `docs/superpowers/specs/2026-06-29-admin-idiomatic-re-expression-design.md` (Phase 1 section, the five de-customization rules, the verification section). The feature is governed by `docs/superpowers/specs/2026-06-29-cairn-tag-management-design.md` (Component 2, the seed, Component 5). This plan is the reconciliation of tag-management Plan 3 Tasks 2–5 with the new idiom.
- **Builds on Phase 0 (do not reshape it).** The frozen developer-facing role interface is the named `@utility` classes `text-muted` / `text-subtle` in `scripts/admin-css.input.css` (guaranteed-value branch, resolving to the Tier-2 `--color-muted` / `--color-subtle` vars). The pilot consumes this interface; it never redefines or extends it. The walled Tier-2 home in `src/lib/components/cairn-admin.css` is fixed. See the ledger `docs/internal/design/2026-06-29-custom-surface-ledger.md`.
- **Builds on Plan 3 Task 1 (landed `9ba8791`).** Consumes the route pair `vocabularyLoad`/`vocabularySave` and `interface VocabularyLoadData { vocabulary: VocabularyEntry[]; usage: Record<string, number>; unlisted: { value: string; count: number }[] }` (exported from `content-routes.ts`), already wired through `admin-dispatch.ts` and `cairn-admin.ts` (the `vocabulary` view, the `saveVocabulary` action). This plan adds only the screen, its mount, the showcase surfaces, and the docs. Do not change the route contract.
- **The five de-customization rules bind every markup line.** Most-load-bearing for this pilot: **Rule 2 — a named role, a themed value.** For secondary text a component writes `text-muted` / `text-subtle` (the role), NEVER the **retired** `text-[var(--color-muted)]` / `text-[var(--color-subtle)]` bracket forms, never the long `[color:var(--color-muted)]` form, never an inline `style="…var(--color-muted)…"`, and never a bare `/60` magic-number opacity for secondary text. The retired forms are exactly what the gate counts (`var(--color-(muted|subtle))` inside brackets or an inline style; `scripts/check-custom-surface.mjs`). The **sanctioned Tier-2 bracket tokens stay allowed and are the established idiom** the near-twin uses: the AA inks `text-[var(--cairn-error-ink)]` / `text-[var(--cairn-warning-ink)]` / `text-[var(--color-positive-ink)]` for an error/warning/guard message (these are Tier-2 with no named role utility, so the bracket form is correct, and the ledger inks exist precisely to keep that text at AA), and the elevation `border-[var(--cairn-card-border)]` hairline. Surface tints over `base-content` (`bg-base-content/[0.06]`, `bg-primary/10`) are likewise sanctioned. The budget-unchanged claim below rides on the muted/subtle retired forms only, not on a zero-bracket screen. **Rule 3** — adopt a native primitive only where 5.6 ships one; the segmented / check-and-tint control and card-selectable states have no native equivalent and stay the shipped hand-rolled recipes. **Rule 1** — never duplicate a theme value into the component.
- **`check:custom-surface` is a standing gate on every code task.** The new screen is a fresh surface, so it must add ZERO to the admin `retiredTokenBudget` (currently 235): no new **retired** muted/subtle bracket reference in `VocabularyAdmin.svelte` or in the `CairnAdminShell` sidebar edit. (The sidebar `{ label: 'Tags', … }` literal carries no token, and the sanctioned inks above do not count toward the budget.) It must add no new `@layer components` rule (the `componentsLayerCap` is 14) and no new unlayered scoped rule (the allowlist is the 6 whole-selector-pinned rules in `scripts/custom-surface-budget.json`: the two theme roots, the two box-sizing resets, the menu-focus ring, and the guarded-button rule). Achieve the look with primitives, the role utilities, and the sanctioned inks; if a scoped override seems unavoidable, that is a ledger + allowlist change requiring a charter-premise note — escalate to the main loop rather than adding one silently. `npm run check:custom-surface` (which self-runs `npm run package` first, per `package.json`) is the authoritative day-one-idiom proof; run it as part of each code task's gate, and treat the component-test markup assertion as a secondary tripwire, not a substitute (the budget gate scans the `.svelte` source, the test scans rendered DOM — genuinely independent checks).
- **Admin design system.** Before any `/admin` component work follow `docs/internal/admin-design-system.md`: `data-theme` goes on a bare wrapper never a styled element, scoped overrides live in `@layer components` (and per the constraint above, the pilot adds none), the Warm Stone tokens, the recipes (cards, eyebrow groups, empty states, the CTA, the command palette), the voice. Reuse `CsrfField`, the hidden-JSON-field post shape, and the `untrack`-seeded `$state` working-copy pattern from `CairnTidySettings.svelte`.
- **Delete is gated cross-branch, failing closed (already enforced in the landed route).** `vocabularySave` rejects removing an in-use `value` per `buildTagUsageIndex(..., { strict: true })` (main ∪ open `cairn/*` branches). The screen reflects this: it disables (or guards) the delete control for an in-use tag and never offers an action the route will reject. `value` is immutable; rename edits `label` only; add derives a slug `value` matching `SAFE_TAG_VALUE = /^[a-z0-9]+(?:-[a-z0-9]+)*$/` and rejects a collision.
- **The seed is a UX affordance, not a safety gate.** Plan 2's per-entry union already prevents any build delist. The screen surfaces the `unlisted` set (in use, not in the vocabulary) and offers to add each, so an editor populates the vocabulary without orphan-flag clutter. Add no separate enable toggle; enforcement is on whenever the vocabulary is non-empty.
- **Two recorded deviations from the cited tag-management spec (provenance, so they are known-stale, not silently contradicted).** (1) **Mount.** The tag-management spec's Component 2 places the screen on the `adminNav` custom-screen seam; this plan mounts it as a first-class **engine** view in the `CairnAdmin.svelte` switch, because taxonomy management is cairn's own job (charter), and the landed route (`9ba8791`) already reserves `vocabulary` and resolves it as an engine view. The `adminNav` seam is for site-owned routes and `normalizeAdminNav` rejects an engine-claimed href. The spec's Component-2 placement line is superseded. (2) **Delete presentation.** Component 2 asks the screen to surface the usage count AND the blocking entries; the landed `VocabularyLoadData` carries `usage` as a per-value count only, with no entry identities. This plan deliberately presents **count-only** (the in-use count plus the guarded delete), the lean charter-correct surface: an in-use tag cannot be deleted at all, so the editor needs to know that and by how much, not necessarily where, and the `unlisted` seed already surfaces identities by value. Naming the blocking posts would reopen the route contract for a heavier load; that is out of scope for the pilot and recorded here as a deliberate deviation, not an oversight.
- **Gate before done (each code task):** the task's targeted test, then `npm run check` (svelte-check 0/0), then `npm test` (exit 0, not just a passing count), then `npm run check:custom-surface` (PASS both trees). Run `npm run package` before `npm test` if a dist import flakes and before any surface/reference check. The showcase e2e and visual baseline are their own gate (Task 3). `npm run check:comments` (the TSDoc + em-dash gate) runs at pass end.
- **Watch items (carried from Phase 0).** (1) The first CI showcase-e2e may need the visual baselines regenerated on the CI image if headless rendering drifts; a baseline update is plan-sanctioned and is the reviewed record of intended drift. (2) Tailwind v4's automatic content detection scans the whole tracked tree (the admin `@source` is additive, not exclusive), so an arbitrary-value class written in doc prose with a non-value placeholder inside the brackets (an ASCII triple-dot or a `*`) compiles to malformed CSS and breaks `npm run package`. Always write the concrete token in any doc, e.g. the real `text-[var(--color-muted)]`, never a placeholder form (memory `tailwind-scans-docs-bad-candidate`).
