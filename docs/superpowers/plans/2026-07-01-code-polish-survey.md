# Code-polish idiom survey: evidence summary (2026-07-01)

Eleven read-only subsystem catalogues from the Phase A survey workflow (raw structured data in the session scratchpad; this summary lists every divergence the charter synthesis must resolve). Task 2 of `2026-07-01-code-polish-pass.md`.


## content

**Subsystem note:** content/ is the healthiest subsystem I'd expect from a young, disciplined rebuild-era codebase: a strong, mostly-followed house style (cairn:-prefixed throw messages for declaration-time errors, ValidationResult for save-time errors, // cairn-cms: headers, verbNoun computers, is-prefixed booleans). The single biggest inconsistency is not a style fork but genuine code duplication: three files (advisories.ts, reference-index.ts, tag-usage-index.ts) independently implement the same ~90-line cross-branch Promise.all index-builder shape, including three separately-declared identical push(index,key,entry) helpers; this is the strongest reuse/simplification candidate in the subsystem, and the one place a shared abstraction (not just a style pick) would pay off. Secondary, smaller items: two files (pending.ts, fields.ts) skip the otherwise-universal `// cairn-cms:` header convention; a bare-noun-vs-verbNoun naming fork exists for computed-value functions (permalink/entryIdentity/tagUsage/formValues vs the dominant deriveX/resolveX/extractX); and one boolean (altIsEdited) inverts the is-prefix convention.
